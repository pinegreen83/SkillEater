Skill Eater 프로젝트 지침 (Working Agreement)

0. 문서 목적

이 문서는 Skill Eater 데모 프로젝트를 진행하는 방식 자체를 정의한다.
게임 기획이 아니라,
**“어떻게 개발하고, 어떻게 질문하고, 어떻게 멈출 것인가”**에 대한 규칙이다.

⸻

1. 프로젝트 목표

1.1 최종 목표
- Steam에 배포된 데모 게임 1개 완성
- SkillMaker 기반 사고를 활용한
- 경험 기반 스킬 습득 시스템
- 내부 시스템 설계 역량
- 실제 배포 경험

을 포트폴리오로 남긴다.

1.2 범위
- 본 프로젝트는 데모까지를 목표로 한다.
- 유료 판매 / 정식 출시 / 장기 운영은 본 프로젝트 범위가 아니다.

⸻

2. 질문 및 작업 범위 규칙

2.1 이 프로젝트에서만 다룰 것

아래 항목은 반드시 이 대화/프로젝트 컨텍스트에서만 진행한다.
- 티켓 발행
- 시스템 설계 논의
- 구현 방향 검증
- 코드 리뷰
- 리팩토링 판단
- 데모 범위 조정

👉 다른 대화로 흩어지지 않는다.

⸻

2.2 코드 제공 규칙
- 코드는 내가 요청했을 때만 제공
- 기본 원칙:
- 설계 조언
- 구조 피드백
- 판단 기준
위주로 진행한다.
- 코드 리뷰 시:
- 네가 작성한 코드 기준으로만 리뷰
- “이렇게 짜라”가 아니라
- 왜 이 구조가 괜찮은지
- 어디서 위험해질 수 있는지
- 대안이 있다면 어떤 선택지가 있는지
를 설명한다.

⸻

3. 티켓 운영 규칙

3.1 티켓 발행 주체
- 티켓은 반드시 네가 “달라”고 요청했을 때만 발행
- 자동 발행 ❌
- 선행 제안 ❌

⸻

3.2 티켓의 성격
- 하나의 티켓은:
- 2~4시간 내 완료 가능한 단위
- 명확한 완료 조건(DoD)을 가진다.
- 티켓은 항상 아래 중 하나에 속해야 한다.
- 시스템 구현
- 시스템 검증
- 시스템 정리(문서/리팩토링)

⸻

3.3 티켓의 기준 문서
- 모든 티켓은 다음 문서를 기준으로 한다.
- System_Design.md
- Project_Guidelines.md
- 티켓이 이 문서와 충돌하면:
- 티켓이 잘못된 것
- 문서를 먼저 수정하지 않는 한, 구현하지 않는다.

⸻

4. 기획 변경 규칙 (스코프 보호)

4.1 기획 변경 금지 원칙
- 데모 범위에서:
- 새로운 시스템 추가 ❌
- 새로운 자원 추가 ❌
- 새로운 성장 구조 추가 ❌

⸻

4.2 변경이 필요할 때
- 모든 아이디어는 즉시 구현하지 않는다.
- 반드시:
- Backlog.md에 기록
- 데모 완료 이후 판단

⸻

5. 설계 결정에 대한 태도

5.1 “정답”을 찾지 않는다
- 이 프로젝트의 목적은:
- 가장 완벽한 시스템 ❌
- 가장 재미있는 게임 ❌
- 대신:
- 의도와 일관성이 있는 시스템
- 설명 가능한 설계
- 스스로 통제 가능한 규모

⸻

5.2 일부 설계는 의도적으로 미확정

다음 영역은 티켓 단위로 실험한다.
- 스킬 경험치 산정 방식
- 피격 기반 습득의 세부 조건
- 자원 소비 밸런스

👉 미정 ≠ 미숙
👉 통제된 실험 상태

⸻

6. 데모 완성의 기준 (Definition of Done)

데모는 다음을 만족하면 “완성”이다.
- 플레이 시작 → 엔딩까지 끊김 없이 진행 가능
- 스킬 습득 → 장착 → 사용 루프가 최소 1회 이상 성립
- 보스 처치 후:
- 다음 지역이 보이고
- 진입은 불가한 상태로 종료
- Steam 데모 빌드 업로드 완료

⸻

7. 이 프로젝트의 핵심 원칙 (한 줄 요약)

이 프로젝트는
“잘 만들기”보다
“끝까지 통제하며 완성하기”를 목표로 한다.
